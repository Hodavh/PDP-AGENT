How the Audit Works

Each product page is scored across 9 dimensions using a Reflexion loop — an Actor LLM produces the audit, an Evaluator checks it against 4 quality criteria, and a Reflector generates a targeted critique if any criteria fail. A second pass corrects the output before the Rewriter produces the page copy. All dimensions are evaluated against absolute best-practice standards from Baymard Institute, NNG, CXL, ASA CAP Code, and MHRA guidance.

System Architecture

End-to-end design of the PW PDP Optimisation Agent

Request Flow

A user submits a product page URL from the browser. The FastAPI backend immediately returns a run_id and spawns a daemon thread so the pipeline never blocks the server event loop. The browser polls GET /api/audit/:id/status every second, receiving the current stage name, until the run reaches complete or error. The full result is fetched in a single call and rendered in the dashboard. On completion the LangSmith metrics cache is automatically invalidated so the Observability Dashboard reflects the new run immediately.

Stage 1 — Scraping

Firecrawl renders the page in a headless browser and extracts ~25 structured fields via an LLM-backed schema: product name, H1, meta title, price, per-serving cost, benefits, ingredients, reviews, trust signals, compliance-sensitive sentences, and FAQ items. A post-processing guard detects when the scraper returns the meta title in the H1 field (a common CDN-rendered page artefact) and corrects the split automatically. A 390×844 px above-the-fold screenshot is captured and the first three product images are fetched with browser-like headers (including Referer) and base64-encoded for direct injection into the LLM prompt. Tavily performs a site-specific search as fallback if Firecrawl fails or returns empty content.

Stage 2 — Reflexion Audit Loop (concurrent)

The audit runs as a Reflexion loop via LangChain's ChatGoogleGenerativeAI (Gemini 2.5 Flash), enabling automatic token reporting to LangSmith. The Actor receives the full page data, images, and a ~2,000-token rubric across 9 conversion-critical dimensions, producing a structured JSON audit with per-dimension scores (1–5), element-level pass/fail checks, compliance flags, and ranked recommendations sorted by impact then effort. The Evaluator (also Gemini 2.5 Flash) checks every recommendation against four binary criteria: grounded in page evidence, specific to this PDP, compliant language, and justified impact score — including a special guard against ATF visibility checks that quote structured data instead of the screenshot. If any compliance criterion fails or more than three criteria fail, the pure-Python Reflector generates targeted critique messages (no LLM call) and the Actor runs a corrective second pass. The Rewriter (Claude Sonnet 4.6 via Anthropic SDK) starts concurrently with the reflexion loop as soon as Pass 1 completes, running in a separate thread via a threading.Event signal — saving 30–120 s on single-pass audits.

Stage 3 — Rewrite (Claude Sonnet 4.6)

Claude Sonnet 4.6 generates a full page rewrite: meta title, H1, sub-headline, benefit highlights, objection-handling FAQ, and trust signal block. The system prompt injects concrete Protein Works brand voice examples (short sentences, scepticism acknowledgement, specific numbers) alongside the GB NHC Register permitted claims list and prohibited language patterns. The anti-hallucination rules explicitly forbid inventing gram amounts, serving counts, or derived prices not present verbatim in the scraped data. Every compliance-sensitive sentence is automatically flagged [HUMAN REVIEW REQUIRED]. If Pass 2 was triggered, the rewriter re-runs with the corrected audit.

Token Tracking & Observability

Every pipeline run is traced end-to-end in LangSmith via @traceable decorators: reflexion_loop → actor_audit → evaluator, with the rewriter as a sibling span. Because the pipeline uses ThreadPoolExecutor, each worker thread calls set_current_run(run_id) at start so a thread-local accumulator (utils/token_store.py) captures Gemini and Claude token counts separately and stores them in scores_json._tokens in SQLite on completion. The Observability Dashboard reads token trends from SQLite (not LangSmith) for accuracy. LangSmith endpoints use per-key TTL caches (embed-url: 1 h, metrics: 5 min, recent-runs: 2 min), pre-warmed in parallel threads 4 s after server startup.
